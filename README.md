**How I think about IT**
- I'd rather understand *why* it works and how it runs under the hood than just *that* it works.
- I start from clean, structured architecture — easy to begin that way, and everything slots into place later.
- Seeing the whole design is also where you spot the holes — security and edge cases come from understanding the system, not patching it afterwards.
- Clean and lean over "good enough" — no needless processes or moving parts.
- I value **simplicity** — like Apple's philosophy, the best solution (and the best design) is the simplest one.
- Think in edge cases: the unlikely input is still a possible input.
- A perfectionist on my own projects — but I know when to ship first and polish toward the ideal after.
- In the AI era the real skill is steering it — architecture, sharp prompts, catching its mistakes.
